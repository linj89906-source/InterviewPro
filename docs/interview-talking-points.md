# 面试讲解稿 — AI 面试助手 (InterviewPro)

> 面试中讲解项目时的完整话术框架。按 STAR 法则组织，约 8-10 分钟。

---

## 1. 项目概述（30 秒）

> "我做的是一个叫 InterviewPro 的 AI 面试助手，面向计算机应届生的求职场景。核心是用 **LangGraph 多 Agent 架构** 把简历分析、模拟面试、技术问答、住宿推荐四个场景整合到一个系统里。整个项目从 Agent 设计到后端 API 到前端页面都是我一个人完成的。"

**面试官听到的关键词**：LangGraph、Multi-Agent、全栈

---

## 2. 背景 & 动机（1 分钟）

> "做这个项目有两个出发点。
>
> 第一，**Agent 是 2025 年 AI 应用最重要的范式**。LangChain 和 LangGraph 从 Chain 进化到 Agent，核心变化是让 LLM 从'被动的推理引擎'变成'主动的决策者'。我想通过实际项目来理解这套范式。
>
> 第二，我身边很多同学找工作时遇到的痛点是分散的——写简历、刷题、模拟面试、找住宿，没有一个工具能一站式解决。所以我决定用 Agent 架构做一个整合方案。"

**面试官听到的关键词**：Agent 范式理解、用户痛点驱动

---

## 3. 架构设计（2 分钟）— 核心部分

```
用户消息 → Supervisor → 条件路由 → 子 Agent → LLM + 工具
```

> "这是整个系统的核心架构。我用 **LangGraph 的 StateGraph** 来编排多 Agent 协作。
>
> **Supervisor 节点**负责意图分类。我设计了**两层分类策略**：
> - 第一层是**中英双语关键词正则匹配**，覆盖了 200+ 个技术关键词，命中率 90% 以上，完全零 LLM 成本
> - 第二层是 **LLM 分类后备**，当关键词匹配不到时才会调用 LLM
>
> 分类完成后通过 LangGraph 的 `add_conditional_edges` 实现条件路由，把消息分发到四个子 Agent。"

> "四个子 Agent 各司其职：
>
> **Coding Agent** 做技术问答，核心是 **RAG**。我用 SQLite FTS5 建了全文检索引擎，覆盖 10 个技术领域。检索到的文档作为上下文注入 System Prompt，让 LLM 基于知识库回答，答案带参考来源。
>
> **Resume Agent** 做简历分析，核心是**结构化输出**。用 LangChain 的 `with_structured_output` 绑定 Pydantic Schema，强制 LLM 输出 7 维度的结构化 JSON。还做了三层 JSON 提取兜底——structured output 失败就 raw text 提取，再失败就加提示重试。
>
> **Interview Agent** 做模拟面试，核心是**状态机**。用结构化 JSON 协议驱动对话状态——question → feedback → follow_up → summary 四个阶段。
>
> **Location Agent** 做住宿推荐，核心是 **Tool Calling**。LLM 从用户输入中提取地点参数，调用高德地图 API 进行地理编码和 POI 周边搜索，再把结果格式化输出。"

**面试官听到的关键词**：LangGraph、StateGraph、条件路由、RAG、FTS5、Structured Output、Tool Calling

---

## 4. 深入细节（2 分钟）— 按面试官兴趣选择

### 如果问：为什么用关键词分类而不是 LLM 分类？

> "有两个原因。第一是**成本**，每个用户消息都调用 LLM 做分类，积累起来很可观。第二是**延迟**，关键词匹配是毫秒级的，LLM 调用是秒级的。我的做法是关键词覆盖常见 case，LLM 做长尾兜底。实测下来关键词命中率 90%+，整体分类准确率很高。"

### 如果问：FTS5 和向量数据库的选择？

> "对于这个项目的规模，FTS5 是更务实的选择。向量数据库需要额外的 embedding 成本和管理复杂度，而 FTS5 是 SQLite 内置的，零部署成本。对于关键词匹配型的技术知识库来说，全文检索的精度完全够用。如果知识库扩展到上万条，我会考虑引入向量检索做混合召回。"

### 如果问：Tool Calling 的实现细节？

> "Location Agent 不是用 LangChain 的 Tool 抽象，而是**手动实现了 Tool Calling 模式**。流程是：LLM 先提取用户输入中的地点信息（城市、地址、公司名）→ 把提取结果传给高德地图 API → 拿到 POI 数据后再交给 LLM 格式化推荐。这样做的好处是我对每一步有完全控制权，也更容易做容错和调试。"

### 如果问：结构化输出的可靠性如何保证？

> "面试官问得好。结构化输出是 AI 应用落地的关键挑战之一。我设计了三层兜底机制：第一层用 LangChain 的 with_structured_output，它会在 API 调用层面约束输出格式。如果这层失败，第二层从 raw text 中用括号匹配提取 JSON。如果还失败，第三层在 prompt 里追加格式强调后重试。只要有一层成功，用户就能拿到结构化结果。这个容错设计在简历分析这种复杂 JSON Schema 场景下特别重要。"

**面试官听到的关键词**：成本意识、工程权衡、容错设计、务实选型

---

## 5. 难点 & 收获（1.5 分钟）

> "最大的难点是**编码问题**。Python 文件、HTTP 请求、PowerShell 终端三者的默认编码不同，中文消息在管道中会出现乱码。Supervisor 的关键词匹配完全失效，所有消息都被路由到 general。我通过字节级验证确认文件编码正确，排查出是 HTTP 客户端的编码问题。这个经历让我对 Python 的 Unicode 编码体系有了很深的理解。"

> "最大的收获是**对 Agent 范式的理解进化了**。开始做这个项目前，我以为 Agent 就是 LangChain 调一个 AgentExecutor。做完后我理解了：Agent 的核心不是工具调用，而是**如何定义 Agent 之间的职责边界和通信协议**。Supervisor 和 Worker 怎么分权？什么时候用关键词、什么时候用 LLM？这些设计决策比具体实现更重要。"

---

## 6. 后续规划（30 秒）

> "如果继续做，三个方向：
> 1. 引入**向量数据库**做混合检索，提升 RAG 召回质量
> 2. 用 **LangGraph 的 checkpoint** 实现多轮对话的状态持久化
> 3. 加入 **MCP 协议**让 Agent 能动态发现和调用外部工具，而不是硬编码 API"

**面试官听到的关键词**：MCP、Checkpoint、向量检索、持续学习

## 面试高频追问速查

| 问题 | 回答要点 |
|------|----------|
| LangGraph 和其他方案的区别？ | StateGraph 显式状态管理 vs CrewAI 的隐式状态；条件路由 vs 顺序执行；checkpoint 持久化 |
| 为什么不直接用 GPT-4？ | 成本考量，DeepSeek 性价比高；架构设计与模型解耦，切换只需改 .env |
| 怎么评估 Agent 质量？ | 意图分类准确率（关键词命中率 90%+）；简历评分与人工评估的偏差；用户反馈 |
| MCP 在这个项目怎么用？ | 目前高德 API 是硬编码调用，下一步用 MCP 协议动态注册工具，Agent 自主发现和调用 |
| 最大的工程挑战？ | 结构化输出的可靠性 → 三层 JSON 提取兜底；中文兼容性 → 编码问题排查 |
